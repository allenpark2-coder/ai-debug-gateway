讓 AI 安全地上你的 SigmaStar 板子

用 ai-debug-gateway 讓工程師和 AI 助手共用同一條 console：唯讀診斷全自動、任何會動到板子的指令都先經過你。

核心觀念

AI 很會讀 log、下診斷指令、交叉比對 SDK 和 wiki，但不能讓它直接拿到序列埠——一條 reboot 或亂改 flash 的指令就可能毀掉重現中的問題現場。這套流程的答案是在中間放一個守門的 daemon：

單一擁有者只有 gatewayd 打得開 UART / SSH，人和 AI 都透過它，永遠不搶埠。
唯讀自動、變更要批准白名單內的唯讀指令 AI 自動執行；會改板子狀態的指令變成提案，等人批准。
全程留紀錄每條指令、每次批准都進 audit log；你打的密碼自動遮蔽，不進任何紀錄。

整體架構

開發機一台（AI、SDK、NFS、gateway 全在上面），SigmaStar EVB 一塊，中間 UART 和乙太網路各一條：

部署走 NFS 而不是走 console 傳檔：板子 mount 開發機的 NFS 之後，AI 編好的 bin / .ko 放進去板子立刻看得到，不用燒錄、不用 tftp 一個一個丟。

開發迴圈

  1. 理解——AI 讀 SigmaStar SDK、內部 wiki、GitHub issue，弄清楚要修什麼、板子長什麼樣。
  2. 編譯——在開發機上用 toolchain 交叉編譯，產出 bin 或 kernel module。
  3. 部署——產物丟進 NFS 匯出目錄，板子那頭 /mnt/nfs 馬上拿得到。
  4. 測試——AI 經 gateway 在板子上執行：唯讀的自動跑，會改狀態的等你批准（或在實驗板上開黑名單模式無人值守）。
  5. 分析——AI 收 console 輸出和測試結果，判斷修好沒，回頭改 code 或更新 issue。

權限模型：誰能做什麼

gatewayd 開三～四個 owner-only 的 Unix socket，角色從 socket 層就分開，AI 拿不到「批准」的能力：

Socket誰用能做什麼
attach工程師互動終端：打字直通板子、approve / reject AI 的提案
controlAI看狀態、讀 console 輸出、提案指令（不會執行）
diagnoseAI自動執行白名單唯讀指令（ps、dmesg、cat /proc/...…），要開 --auto-readonly 才存在
unsafeshellAI黑名單模式：無人值守執行變更指令，僅限可重刷的實驗板，要另簽 risk_accepted 才存在

兩種自動模式是獨立開關：日常用白名單就夠；跑整夜迴歸測試、板子隨時可以重燒 IPL + kernel 救回來的，才考慮黑名單模式。OTP / eFuse 這類燒了就回不去的操作，永遠不該給黑名單模式的板子。

快速上手（10 分鐘）

repo 的 share/ 有打好的分享包，解開跑安裝腳本就好：

# 從 repo share/ 抓 tarball，先解開再跑（binary 在包裡）
tar xzf ai-debug-gateway-share_0.1.0_linux_amd64.tar.gz
cd ai-debug-gateway-share_0.1.0_linux_amd64
./setup.sh   # 裝 binary、自動偵測 USB 轉接器、建好 profile 和 policy

之後每天就三行：

gatewayd --auto-readonly   # 終端機 1：daemon，獨占序列埠
gateway start              # 終端機 2：開 session
gateway attach             # 互動終端，打字直通板子

在 attach 裡按 Ctrl-] 進本地命令模式：approve <id> 批准提案、retry uart 斷線重連、detach 離開但保留 session。看到 Password: 時輸入會自動遮蔽；畫面卡在遮蔽狀態就 Ctrl-] 後打 secret-done。

SigmaStar EVB 的 console 通常是 115200 8N1，跟預設 profile 一致；不同的話改 ~/.config/ai-debug-gateway/profiles/default.json 即可。

上線前的注意事項

網路先通。NFS 和 SSH 都吃乙太網路，先確認板子拿得到 IP；順便對時（板端時鐘常停在 1970，會影響 make 時間戳和 TLS）。

「跑測試 bin」不是唯讀。執行 NFS 上的程式一定過不了白名單——要嘛人工 approve，要嘛在實驗板上開黑名單模式。規劃無人值守測試時先想清楚走哪條。

UART 和 SSH 一次用一個。一個 session 一個 transport：日常測試走 SSH（快），UART 留給開機階段和網路掛掉時救援。

已知問題（修復中）：經 UART 的 diagnose 不要跑 dmesg——輸出裡的開機字樣會被誤判成板子重開機；有密碼的板子登入中 status 可能提早顯示 READY；遮蔽視窗卡住時用 secret-done 手動解除。走 SSH 可避開前兩項。
